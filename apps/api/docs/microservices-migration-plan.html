UITFood API Microservices Migration Plan

Incremental plan for migrating the NestJS backend in apps/api from a modular monolith to a microservice architecture without a big-bang rewrite.

Prepared: 2026-06-04 Scope: backend only Current shape: modular monolith Target: service-owned data and contracts
Executive Summary Current State Target State Service Boundaries Contracts Data Ownership Migration Phases Deployment Changes Testing and Observability Risks Checklists Timeline

Executive Summary

Do not split the backend by copying controllers into separate apps first. Split by domain ownership, data ownership, and stable contracts. The current codebase already has the right starting point: bounded contexts under apps/api/src/module, CQRS events in apps/api/src/shared/events, and interface-based ports in apps/api/src/shared/ports.

Recommended strategy: use the strangler pattern. Keep the public API stable through an API gateway, externalize events while the monolith still runs, then extract one bounded context at a time.

Primary Goal

Move from one deployable NestJS process with one unified Drizzle schema to multiple independently deployable services with their own databases or database schemas.

Primary Constraint

Preserve client routes for the mobile app, restaurant web portal, and admin app. Route changes should be hidden behind the gateway.

Primary Risk

Data consistency during checkout, payment confirmation, promotion reservation, and notification fan-out. These paths need explicit idempotency and replayable events.

Current State

The backend is a NestJS 11 application in apps/api. It uses PostgreSQL through Drizzle ORM, Redis, Better Auth, VNPay, Cloudinary, Firebase push, SMTP email, Socket.IO, and OpenTelemetry.

Current Backend Modules

Current module Purpose Migration implication

restaurant-catalog Restaurants, menus, modifiers, delivery zones, search. Strong service candidate; already publishes catalog events.

admin-analytics and ordering/analytics Read-side reporting and operational analytics. Extract late as event-fed read models.

Existing Strengths to Reuse

  • Bounded-context directories already exist under apps/api/src/module.
  • Ordering already uses ACL snapshot tables for catalog data.
  • Notification already has its own restaurant snapshot projection.
  • Cross-context ports already exist for payment, promotion, and review eligibility.
  • Domain events already describe important business transitions.
  • OpenTelemetry and JSON logging are already in place.

Current Coupling to Remove

  • The monolith has one AppModule importing every domain module.
  • apps/api/src/drizzle/schema.ts exports all tables into one unified schema.
  • The CQRS event bus is in-process, so event delivery is not durable across services.
  • Some support-context calls are synchronous in-process ports, not network APIs yet.
  • Shared auth/session behavior is process-local and must be moved to gateway claims.

Target State

The target state keeps the pnpm monorepo but creates separate service apps. Each service has its own module root, configuration, Drizzle migrations, deployment artifact, and telemetry service name. 

apps/
  api-gateway/             # Public HTTP API and auth/session edge
  identity-service/        # Better Auth, users, roles, approvals
  catalog-service/         # Restaurants, menu, modifiers, zones, search
  ordering-service/        # Cart, checkout, lifecycle, history, ACL snapshots
  payment-service/         # VNPay, transactions, refunds, payment timeout
  promotion-service/       # Promotions, coupons, usage reservation
  notification-service/    # Notifications, push/email, WebSocket gateway
  media-service/           # Cloudinary image lifecycle and ownership
  review-service/          # Review submission and restaurant ratings
  analytics-service/       # Admin and restaurant read-side analytics
packages/
  contracts/               # Event payloads, DTOs, OpenAPI schemas, shared enums
  observability/           # Shared OTel/logger helpers, optional
  database/                # Shared Drizzle helper only, no shared tables
Monorepo decision: keep the monorepo during the migration. A polyrepo split should only happen after service boundaries are stable and CI/CD can deploy services independently.

Target Communication Model

Pattern Use for Rule
Gateway routing Public client requests from mobile, web, and admin. Clients keep calling /api/*; gateway routes internally.
Durable domain events State changes such as order placed, payment confirmed, menu updated. Use outbox/inbox and idempotent consumers.
Internal request/response API Immediate business decisions such as payment initiation or promotion reservation. Use timeouts, idempotency keys, retries only for safe operations.
Local read models Data a service needs from another owner, such as catalog snapshots in ordering. No cross-service database joins.

Contracts

Event Envelope

Before extracting services, wrap all current domain event payloads in a durable event envelope. This makes replay, tracing, versioning, and idempotency possible. 

{
  "eventId": "uuid",
  "eventType": "OrderPlaced",
  "schemaVersion": 1,
  "producer": "ordering-service",
  "occurredAt": "2026-06-04T00:00:00.000Z",
  "correlationId": "request-id-or-trace-id",
  "causationId": "previous-event-id-or-command-id",
  "idempotencyKey": "business-key",
  "payload": {}
}

Data Ownership

Start with one PostgreSQL instance if needed, but separate ownership by schema and database user. Move to separate databases once services have proven stable in production.

Service Owned data Can expose to others through
Identity Auth tables from auth.schema.ts, users, sessions, roles. Gateway-authenticated claims, internal identity API.
Catalog Restaurants, delivery zones, menu categories, menu items, modifiers. Catalog APIs and catalog events.
Ordering Orders, order items, status logs, carts in Redis, catalog snapshot tables. Order APIs and order events.
Payment Payment transactions, refund state, VNPay identifiers. Payment APIs and payment events.
Promotion Promotions, coupon codes, promotion usage/reservation rows. Promotion APIs and usage events.
Notification Notifications, preferences, delivery logs, device tokens, restaurant snapshots. Notification APIs and WebSocket messages.
Media Image records, Cloudinary metadata, owner references. Media APIs and cleanup events.

Analytics Read-side aggregate tables and materialized views. Analytics APIs only.
Data rule: once a table is assigned to a service, other services must not read or write it directly. If another service needs that data, it gets a local projection from events or calls the owning service.

Migration Phases

Phase 0: Baseline and Boundary Audit

Duration: 1-2 weeks

  • Inventory every public route, controller, table, scheduled task, event, and external dependency.
  • Mark each table with an owning target service.
  • Mark each current cross-module import as allowed, temporary, or forbidden.
  • Fix known event gaps before extraction, especially catalog approval/unapproval event coverage.
  • Generate and archive current OpenAPI output from /api-spec.json as the client compatibility baseline.
  • Capture baseline metrics: p95 latency, checkout success rate, VNPay IPN success, notification delivery, e2e pass rate.

Exit criteria: service ownership map approved, route baseline captured, and no unknown critical dependency in checkout/payment paths.

Phase 1: Contract-First Hardening Inside the Monolith

Duration: 2-3 weeks

  • Create packages/contracts for event payloads, shared DTOs, error codes, route contracts, and shared enums.
  • Move event classes from apps/api/src/shared/events into versioned contract files.
  • Add event envelopes with eventId, schemaVersion, producer, correlationId, and idempotencyKey.
  • Add outbox tables per future service owner while still using the monolith process.
  • Add inbox/processed-event tables for idempotent event consumers.
  • Convert remaining cross-context implementation imports to ports or event projections.
  • Add contract tests for payment, promotion, review eligibility, and catalog snapshot events.

Exit criteria: events and ports are versioned contracts, consumers are idempotent, and monolith behavior is unchanged.

Phase 2: Introduce Broker and Gateway Without Extracting Services

Events Gateway

  • Add a durable message broker. Prefer a production-grade broker such as NATS JetStream or RabbitMQ. Redis Streams is acceptable as a short-term bridge because Redis is already present, but avoid non-durable pub/sub for business events.
  • Publish existing domain events to the broker through the outbox relay.
  • Keep current in-process event handlers running until external consumers are ready.
  • Create apps/api-gateway and route all public traffic to the current monolith first.
  • Move request ID, CORS, auth/session edge handling, rate limits, and public API docs aggregation to the gateway.
  • Add service-to-service auth using internal tokens or mTLS-compatible headers.

Exit criteria: clients can call the gateway with no API behavior change, and business events are visible in the broker.

Phase 3: Extract Notification Service First

Recommended first extraction

  • Create apps/notification-service using the current notification module code.
  • Move notification tables and notification restaurant snapshot tables into a notification-owned schema.
  • Consume catalog, order, payment, and review events from the broker.
  • Move the Socket.IO notification gateway into the notification service.
  • Route /api/notifications and notification WebSocket traffic through the API gateway.
  • Run shadow mode first: external service writes to its own database while monolith notification remains authoritative.
  • Compare notification rows, delivery logs, WebSocket delivery, and retry behavior before cutover.

Exit criteria: notification service handles production traffic, monolith notification handlers are disabled, and delivery metrics remain stable.

Phase 4: Extract Catalog and Media Services

Catalog Media

  • Create apps/catalog-service for restaurants, menu items, categories, modifiers, zones, and search.
  • Create apps/media-service for image records and Cloudinary operations, or keep media as an internal catalog adapter until the image boundary is clean.
  • Backfill ordering and notification snapshots from the new catalog service.
  • Publish catalog events from the catalog service only.
  • Route /api/restaurants, /api/menu-items, /api/search, and delivery zone routes through the gateway.
  • Remove direct catalog table reads from non-catalog services.

Exit criteria: catalog writes happen only in catalog service, ordering snapshots stay current from broker events, and public catalog routes match the baseline OpenAPI behavior.

Phase 5: Extract Promotion Service

Synchronous support service

  • Create apps/promotion-service from the existing promotion module.
  • Expose internal APIs for preview, reserve, confirm, and rollback.
  • Require idempotency keys on reservation, confirmation, and rollback operations.
  • Route public promotion routes through the gateway.
  • Update ordering to call the promotion service over the network instead of the in-process port implementation.
  • Keep the same port interface in ordering so domain code does not care whether the adapter is local or remote.

Exit criteria: checkout promotion behavior, reservation rollback, and admin/restaurant promotion routes pass contract and e2e tests.

Phase 6: Extract Payment Service

VNPay critical path

  • Create apps/payment-service from the payment module.
  • Move payment transaction tables and timeout task to payment-owned storage.
  • Expose internal payment initiation and mark-failed APIs.
  • Publish PaymentConfirmedEvent and PaymentFailedEvent through the broker.
  • Consume OrderCancelledAfterPaymentEvent for refund workflow.
  • Keep VNPay IPN routes stable through the gateway and preserve signature verification behavior.

Exit criteria: VNPay happy path, amount mismatch, timeout, duplicate IPN, and cancellation-after-payment tests pass against the remote payment service.

Phase 7: Extract Ordering Service

Core domain Highest risk

  • Create apps/ordering-service for cart, checkout, lifecycle, history, order eligibility, and ordering analytics if not yet moved.
  • Move order tables, order snapshot tables, app settings, and cart Redis keys to ordering ownership.
  • Update ordering adapters to call remote payment and promotion services.
  • Consume catalog events for restaurant, menu item, and delivery zone snapshots.
  • Publish order events through the broker only after local transaction commit.
  • Route /api/carts, /api/orders, and role-specific order routes through the gateway.
  • Run a controlled cutover by route family: cart first, then read-only order history, then checkout, then lifecycle transitions.

Exit criteria: checkout, order lifecycle, payment integration, promotion rollback, order history, and notification fan-out remain consistent under load.

Phase 8: Extract Review and Analytics Services

Late extraction

  • Create apps/review-service and call ordering eligibility through an internal API.
  • Publish ReviewSubmittedEvent through the broker.
  • Create apps/analytics-service for admin and restaurant analytics.
  • Build analytics as event-fed read models instead of live joins across service data.
  • Route review and analytics routes through the gateway.

Exit criteria: review submission, notification fan-out, and analytics dashboards work without direct cross-service database access.

Phase 9: Data Split and Monolith Decommission

Finalization

  • Remove the unified apps/api/src/drizzle/schema.ts dependency from extracted services.
  • Give each service its own Drizzle config, migrations, and database credentials.
  • Run read-only compatibility checks between old monolith tables and new service-owned tables.
  • Turn off monolith routes after gateway traffic has been migrated.
  • Keep a rollback window with database snapshots and route-level gateway fallback where feasible.
  • Delete dead modules from apps/api only after no production traffic or scheduled task depends on them.

Exit criteria: the old monolith is no longer a required production dependency.

Deployment Changes

Local Development

  • Extend docker-compose.dev.yml with broker, gateway, and extracted services as they are introduced.
  • Keep one command to start the full stack, but support service-specific dev scripts.
  • Use separate ports only internally; public local clients should still target the gateway.

Monorepo Build and Scripts

  • Add each service as an apps/* workspace package.
  • Keep Turborepo tasks for build, test, typecheck, lint, and dev.
  • Add service-level scripts such as pnpm --filter ordering-service test.
  • Move generated contracts into packages/contracts so service builds depend on contract changes.

Render and Infrastructure

  • Extend infra/render/main.tf from one API web service to multiple web/private services.
  • Keep only the gateway public. Domain services should be private/internal where the platform supports it.
  • Create separate environment groups per service to avoid leaking VNPay, SMTP, Firebase, or Cloudinary secrets to unrelated services.
  • Expose separate health checks such as /health/live and /health/ready.
  • Set distinct OTEL_SERVICE_NAME values for each service.

Testing and Observability

Required Test Layers

Layer Purpose Examples
Unit tests Keep current domain behavior stable inside each service. Order lifecycle transitions, promotion engine, VNPay handlers.
Contract tests Prevent breaking changes in events and internal APIs. Payment initiation API, promotion reservation API, catalog event payloads.
Consumer-driven event tests Ensure each consumer can process producer event versions. Notification consumes OrderStatusChanged; ordering consumes MenuItemUpdated.
Integration tests Verify service plus database plus broker. Outbox relay, inbox idempotency, retry and dead-letter behavior.
E2E tests Protect client-visible workflows through the gateway. Checkout, VNPay return/IPN, order tracking, push token registration, review submission.
Load tests Detect latency regressions from network hops. Use existing tools/k6/realistic-user.js as the starting point.

Observability Requirements

  • Propagate trace context from gateway to every service and broker event.
  • Log correlationId, eventId, orderId, and userId where applicable.
  • Emit RED metrics per service: request rate, error rate, duration.
  • Emit broker metrics: outbox lag, consumer lag, dead-letter count, retry count.
  • Add business metrics: checkout success rate, payment confirmation delay, notification delivery rate, promotion rollback count.

Risks and Controls

Risk Why it matters Control
Distributed checkout failure Checkout touches catalog snapshots, promotion reservation, payment initiation, and order persistence. Keep ordering as the orchestrator, use idempotency keys, and add compensation for promotion/payment failures.
Event loss or duplicate delivery Notification, analytics, and snapshots depend on events. Use outbox/inbox, durable broker, idempotent handlers, dead-letter queues, and replay tooling.
Data drift in snapshots Ordering must not use stale menu price or restaurant availability at checkout. Backfill snapshots, monitor snapshot lag, reject checkout when required snapshot data is missing or stale.
Auth inconsistency Services must agree on user identity and role claims. Validate sessions at gateway and forward signed claims. Internal services trust only gateway-signed claims.
Operational complexity More services means more deployments, metrics, logs, secrets, and failures. Extract only when contracts are stable, keep shared observability, and automate service templates.
Frontend breakage Mobile, web, and admin already depend on current routes. Gateway preserves public paths and response shapes; compare against OpenAPI and e2e baselines.

Checklists

Before Extracting Any Service

  • Service owns a clear table list and no other service writes those tables.
  • Public routes are mapped through the gateway.
  • Internal APIs and events are versioned in packages/contracts.
  • Outbox publish and inbox idempotency are implemented.
  • Service has health checks, logs, traces, metrics, and alerts.
  • Contract tests pass for all callers and consumers.
  • Rollback path is documented at route and database level.

Before Cutting Over Traffic

  • Gateway can route a percentage or route family to the new service.
  • Shadow writes or read comparison have run long enough to catch drift.
  • E2E tests pass through the gateway, not through direct service URLs.
  • Broker lag, service latency, and error-rate alerts are active.
  • On-call runbook includes retry, replay, and rollback steps.

After Cutover

  • Disable duplicate monolith handlers for that service boundary.
  • Remove direct table access from old modules.
  • Archive migration notes and contract versions.
  • Track production metrics for at least one full peak traffic cycle.
